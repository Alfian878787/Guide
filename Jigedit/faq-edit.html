FAQ on Publishing Problems (in progress)

Here are some notes to help diagnose a publishing problem with edit.w3.org.

Nearby: Publishing with edit.w3.org

Logs

Logs can be found at the following locations on edith

  • /jigedit/trace.txt (history in directory /jigedit/Traces/)
  • /jigedit/Jigsaw/logs/log_aaaa_mm, (ECLF format) but not real-time updated

Global Issues

In general, we are asked to look at this problem when users got an error trying to publish a document or when the publication seems ok but the modifications are not effective in the server version.

There may be different causes to these problems (non-exhaustive list)

ACL setting (1)

The directory in which the user tries to publish must have permissions set. Otherwise, authentication is refused in the editor and a "401 Unauthorized" message is generated in the logs.

ACL setting (2)

When a new document is created, the acl setting is not immediately propagated to the mirrors, This causes a "403" error when the www.w3.org version is reached (the edit.w3.org version works well). Wait some minutes or change the acl with ,access

(we are currently investigate this problem to find a fix).

Names case-sensitive

The names of the published documents are case-sensitive, that may lead to some confusion (typically 'overview.html' instead of 'Overview.html'), as no errors are reported in this case.

CVS conflict

We recently got some sysreqs about problems in uploading documents, independently of the editor. They resulted in a "409 Conflict" error in the logs, mostly for new resources (msg 'with a cvs status equals to : Locally Added') but also for existing ones (msg 'with a cvs status equals to : Locally Modified').

We are still investigating the cause of this problem. A cvs operation on the resource or using the cvs form (http://edit.w3.org/address/CVS) should fix it.

Editor specific problem

password-in-the-clear with Amaya

A bug in the libwww caused this problem. The patched version of Amaya is the 11.4.7 (previous versions should be replaced).

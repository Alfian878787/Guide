Description

edit.w3.org is another way of editing W3C web space. It is a special purpose HTTP server that allows authenticated users to perform an HTTP/PUT on our servers.
edit.w3.org is a Jigsaw server enhanced with CVS functionalities. With this tool you can use the CVS system through a form interface and therefore edit the W3C web space in a safe way.

Anyone with a W3C account can publish to W3C web space depending on the access control (ACL) of that web space. To check the ACL of a http://www.w3.org URI append ,access to it.
Note not all of www.w3.org web space is open to ACLs and ,access will indicate so.

ACL'd space

Access Control Lists (aka ACL) allow to define flexible and modifiable access control on a given resource (document or directory). Most of the directories in our web site are under ACL, especially the directories in the form of http://www.w3.org/YYYY/ (where YYYY is a 4 digit year).

What does it imply for the publication?

You can edit an ACL'd resource if you have write permission on it, or if you belong to a group that has write permission.
If you are an editor of an ACL'd space, you will be able to update pages. But if you want to create a new resource in this space, you'll have to ask your Team contact to change the ACLs for your new resource otherwise nobody (even yourself) will be able to access it.

The tool ,access (,access appended to an URI) indicates if you can access that URI. In any cases, ask your Team contact if you have troubles with permissions.

How to use edit.w3.org

There are two ways to use edit.w3.org

  • Through a tool that does HTTP PUT and that supports SSL:
    • HTML authoring tools such as BlueGriffon, Amaya, Winie.
      Other authoring tools as SeaMonkey, Kompozer, Arbortext Editor or Oxygen XML for instance also implement HTTP PUT but have not been yet tested with edit.w3.org.
      Note that previous versions of Adobe Dreamweawer didn't implement this method, recent versions may use it, please verify the available features.
    • Transfer data tools such as curl or the libwww PUT tool (no longer maintained).
  • Through Webdav

Setting up your Authoring tool

Among the authoring tools that support HTTP PUT, we recommend using BlueGriffon or Amaya

  • BlueGriffon is a WYSIWYG web editor that supports HTML5, it is powered by Gecko, the rendering engine of Firefox. In order to use the HTTP PUT method, you have to get and install the PUTter add-on.
  • Amaya is a WYSIWYG web editor that natively implements the PUT method but doesn't support HTML5. Note that only the latest version of Amaya (version 11.4.4) supports SSL.

However, if you want to use another authoring tool, you should read this section. Most authoring tools come with default settings that do not apply to edit.w3.org.
One of the important things to check for is how the authoring tool handles images and links.

  • Links: the general policy on our web site is to always use relative links when pointing to resources within W3C's site. Verify that relative links are not transformed into absolute ones when you load a edit.w3.org document. Also check that inserted links into W3C's site are related to document location (verify for instance that the checkbox 'Make URL relative to page location' is checked when you edit a link with BlueGriffon).
  • Images: when creating a document that links to images, the images should be available from our web site, links to them should be made relative to document location and the authoring tool should not try to save them. If new images need to go with the document, verify that the authoring tool proposes to upload them when you publish it, or upload them separately with curl for instance.

How to publish in W3C's web space

Where do I PUT my file ?

It is simply a matter of mapping the actual URI of the resource into a edit.w3.org-based URI.
Our www.w3.org servers are set to redirect PUT requests to the corresponding edit.w3.org URI but if your client cannot handle redirect simply change www to edit in the URI.

How do I put HTML files?

There are basically two methods, depending on what tool you use:

  • With an authoring tool that have HTTP PUT implemented (which is mandatory to use edit.w3.org, see How to use edit.w3.org):
    When the document is ready to be published, hit Save As or its equivalent (it might be Publish) then, enter the correct edit.w3.org location for the document.
  • Using curl to transfer a local file to a remote URI is quite simple, you have to use the options '--upload-file' (or '-T') and '--user' as follow:
    curl --user name:password --upload-file source_file remote_uri

What about other types of files ?

To PUT any type of file (expect php files), you can use curl. As explained above, it takes a source file and a destination URI, and performs a PUT.
Team members (and only them for the moment) can also use webdav.

Any type of document whose mime-type is recognized by edit.w3.org can be transferred on edit.w3.org.

What about directories ?

Directories are automatically created and added to our CVS repository when needed.

Using edit.w3.org through WebDAV

WebDAV is an HTTP-based protocol that allows accessing the content of an HTTP server as a networked filesystems; to put it simply, it allows to edit directly a web site as if it were an additional disk on your computer.

See How to publish with webdav on edit.w3.org
(note that webdav is restricted to Team members for the moment).

edit.w3.org and CVS

edit.w3.org is more than an HTTP server with per-user authentication on PUT. It relies on CVS to provide a history of all the changes. New changes are automatically committed to our CVS repository upon saving.
edit.w3.org is actually a web-based interface to CVS with PUT capabilities. It uses forms to perform the most common CVS commands, and all you need is just a browser.

Basic commands of CVS

You can perform most CVS commands through your browser by calling upon the CVS form within the directory you wish to maintain. Note that this view also allows you to bypass the normal Overview page and see the directory contents. This is convenient for other purposes too!

To access the CVS form in the directory you wish to maintain, simply append "CVS" to the edit.w3.org URI for the directory. Referring to "Where do I PUT my file?" above:

http://edit.w3.org/Guide/CVS

Troubleshooting

  • I get a '415: Unsupported media type' error when trying to PUT a document.

    You were probably trying to PUT a document with an unknown extension. Although lot of mime-types are registered in edit.w3.org (html and htm for text/html, jpg and jpeg for image/jpeg, png for image/png), the extension of the faulty document could be different from the one associated with its mime-type in edit.w3.org, or it could be an unknown mime-type. In this case, contact <webreq@w3.org> to ask for the mime-type to be added.

  • Something went wrong (it hangs, it stopped, it crashed...), what should I do ?

    The first thing to do is to look at the CVS form of the directory where you just tried to PUT. It will show you the status of this directory, and the CVS status of the files in there, including the file you just tried to PUT.

    To access the CVS form for http://www.w3.org/Groups/MyGroup/Draft/my_draft.html, just go to http://edit.w3.org/Groups/MyGroup/Draft/CVS
    There you can see the status of the file you just tried to PUT. If it says 'Conflict', contact <webreq@w3.org>. If it says 'Needs patch', select Update then commit.

  • I've put a new document on edit.w3.org, but when I try to access it on the main server, I get a 401 Forbidden. What's the matter?

    You must be editing in ACL'd space. See the section about it and contact <webreq@w3.org> if it doesn't help.
